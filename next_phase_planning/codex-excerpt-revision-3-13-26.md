# Monkey Mind Excerpt Revision Plan

**Date:** March 13, 2026
**Purpose:** Revise `excerpt.html` so the portfolio packet matches the current proposal, the live improvisation prototype, and the updated Banff application narrative.

## Context

The project narrative has changed materially.

The proposal now frames *Monkey Mind* as:
- an electronic literature work whose key formal move is the transition from stochastic generation to LLM improvisation
- a current project with a live single-voice improvisation prototype
- a Banff residency proposal for expanding that prototype into a multi-model dialogue

The portfolio excerpt still presents the project primarily as:
- a stochastic duet
- a code-and-word-pool packet
- a document centered on the existing two-voice version

That misalignment now matters. The proposal is making stronger claims than the portfolio currently supports.

## Core Principle

The portfolio packet does not need to become a full second proposal.

It does need to show that the project has evolved beyond the current duet and that the improvisation prototype is a real, central part of the work.

## What Has Changed Since The Current Excerpt

These project changes should now be reflected in the portfolio packet:

- The proposal now leads with the formal transition from rule-based generation to LLM improvisation.
- The live improvisation prototype is now available at:
  - `https://curtrode.github.io/monkey_mind/openAI-improv.html`
- The application narrative now treats:
  - the current duet as an existing stage of the work
  - the monologue prototype as the immediate proof of concept
  - the multi-model dialogue as the residency-scale next phase

## Proposal Changes The Excerpt Must Now Support

The latest proposal language makes several claims that the portfolio packet should reinforce:

1. *Monkey Mind* is not only a stochastic duet; it is an evolving work whose dramatic arc is the transition into improvisation.
2. A working single-voice monologue prototype exists now.
3. The prototype demonstrates a staged handoff:
   - stochastic generation
   - echo
   - bend
   - drift
4. The Banff residency work expands this monologue into a multi-model dialogue.
5. The project remains grounded in Montfort's lineage, but it is no longer defined only by that lineage.

If the portfolio packet does not make these points legible, the proposal and portfolio will feel out of sync.

## Main Problems In The Current Excerpt

### Problem 1: Page 1 still frames the project as a stochastic duet

Current effect:
- The opening undersells the new formal argument.
- The improvisation prototype appears nowhere in the opening frame.

Revision direction:
- Reframe page 1 as an evolving project, not only a description of the current duet.

### Problem 2: The packet describes only one live work

Current effect:
- Reviewers are sent to the duet version and the repository, but not clearly to the improvisation prototype.
- The portfolio does not visibly support the proposal's claim that a prototype already exists.

Revision direction:
- Add the live improvisation link on page 1 alongside the duet and repository links.
- If possible, name the two live pieces distinctly:
  - duet version
  - improvisation prototype

### Problem 3: "How to Read This" is now incomplete

Current effect:
- The packet says it contains engine, word pools, and sample output.
- That is no longer a complete account of the project's present form.

Revision direction:
- Update this section so reviewers understand the packet documents one stage of a larger evolving work.

### Problem 4: The packet is already at the 10-page limit

Current effect:
- New material cannot simply be appended.
- Adding a monologue appendix without cuts would make the portfolio noncompliant.

Revision direction:
- Replace and compress rather than append.

## Recommended Revision Strategy

Do not rebuild the packet from scratch.

Instead, keep the existing structure but revise page 1 and make room for one compact monologue/prototype section by cutting lower-value material later in the packet.

## Recommended Page 1 Changes

### 1. Update the subtitle

The current subtitle:
- `Too many tabs open — a stochastic duet`

Problem:
- It accurately describes one stage of the project, but not the project as now presented in the proposal.

Revision direction:
- Keep `Too many tabs open`
- Shift the framing away from "a stochastic duet" alone
- Emphasize evolution, improvisation, or staged transition

### 2. Rewrite the opening description

The first paragraph should now do three things:

1. describe the existing duet briefly
2. identify the transition into improvisation as the project's current formal center
3. position the excerpt as documentation of one phase within a larger evolving work

### 3. Add the improvisation link

Page 1 should now include all of these:
- duet version
- improvisation prototype
- source repository

Optional:
- note that the improvisation prototype is also documented in video form for reviewers without API access

### 4. Revise the Montfort paragraph

Keep the lineage.

But change the claim from:
- carrying stochastic generation into dialogue

To something closer to:
- beginning in that lineage and moving toward a formal transition from combinatory generation to improvisational response

### 5. Rewrite "What follows"

The current language says the packet contains the core generative source of the piece.

That should now be broadened so the packet is presented as:
- an excerpt from an evolving project
- documentation of the stochastic engine and duet structure
- context for the newly launched improvisation phase

## Recommended "How to Read This" Changes

This section should no longer describe only three components mechanically.

It should guide the reviewer through the packet as a progression:

1. **Framing**: the project's current state and trajectory
2. **Engine**: the stochastic system the work begins from
3. **Word Pools**: the duet's poetic vocabulary
4. **Sample Output**: the two-voice stochastic stage
5. **Improvisation Prototype**: the new formal development, documented via link and brief explanation

If page count is too tight, this can be compressed into three items:
- current duet
- stochastic engine
- improvisation next phase

## Recommended Structural Change Inside The Packet

The strongest improvement would be to create a small section near the end titled something like:

- `New Development: Improvisation Prototype`
- `Current Extension: Stochastic to Improvisational Drift`
- `Next Phase of the Work`

That section should include:
- one short paragraph describing the monologue prototype
- one direct link to `openAI-improv.html`
- one sentence explaining staged drift:
  - echo
  - bend
  - drift
- one note that the Banff residency expands this toward multi-model dialogue

## What To Cut To Make Room

Because the packet is already at 10 pages, new material must come from compression.

Best candidates for reduction:

1. Reduce one page of sample output.
2. Compress one page of word-pool display.
3. Shorten repeated explanatory annotations that restate what the arrays already show.

Lowest-value material to preserve at full length:
- multiple full pages of sample output
- exhaustive array presentation if it prevents the improvisation update from appearing

Highest-value material to preserve:
- page 1 framing
- at least one clear engine excerpt
- evidence of the duet's actual language
- explicit documentation of the improvisation prototype

## Recommended New Portfolio Logic

The packet should implicitly present the project in three stages:

1. **Lineage / current base system**
   - rule-based stochastic generation
2. **Current live duet**
   - two voices
   - TTS
   - terminal interface
3. **New improvisation prototype**
   - single-voice monologue
   - staged drift into LLM response
   - foundation for residency-scale multi-model dialogue

This aligns the portfolio with the proposal without turning the excerpt into a second proposal.

## Suggested Tone For New Portfolio Language

The excerpt should sound descriptive and documentary, not argumentative.

Good emphasis:
- this is how the current work operates
- this is how the project has evolved
- this is what the new prototype adds formally

Avoid:
- overloading the packet with precedent discussion
- repeating the entire proposal argument
- adding a large appendix that displaces the work itself
